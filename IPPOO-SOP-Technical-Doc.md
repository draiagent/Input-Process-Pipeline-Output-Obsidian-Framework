## 1. 專案概述

### 1.1 背景與目的

本文件定義一套跨 AI 模型的知識處理工作流，核心問題是：**當使用多個 AI 模型（Gemini、Claude、Claude Code、GPT-5.6）協作處理同一份內容時，如何在模型彼此無法共享記憶的前提下，維持知識的一致性與可追溯性？**

解法不是讓模型互相「記憶」彼此的輸出，而是建立一個 **外部單一事實來源（Single Source of Truth）**——本文件採用 **Obsidian Vault** 作為記憶中樞，所有模型只負責讀寫這個共用知識庫的特定區塊。

### 1.2 命名說明

- **IPPOO**：本框架的五個處理層級縮寫
  - **I**nput（輸入）
  - **P**rocess（處理／驗證）
  - **P**ipeline（自動化管線，選用）
  - **O**utput（視覺化輸出）
  - **O**bsidian（知識中樞／記憶層）
- **SOP**：Standard Operating Procedure，將 IPPOO 落地為可重複執行的操作步驟

## 2. IPPOO 架構總覽

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│    Input    │ ──▶ │   Process   │ ──▶ │  Pipeline   │ ──▶ │   Output    │
│ Gemini /    │     │   Claude    │     │ Claude Code │     │  GPT-5.6    │
│ NotebookLM  │     │  (驗證邏輯)  │     │  (選用自動化) │     │  (視覺產出)  │
└─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
       │                    │                    │                    │
       └────────────────────┴────────────────────┴────────────────────┘
                                     │
                                     ▼
                        ┌─────────────────────────┐
                        │   Obsidian Vault         │
                        │  （雙向連結知識中樞 / SSOT）│
                        └─────────────────────────┘
```

### 2.1 核心設計原則

| 原則 | 說明 |
|---|---|
| **記憶體外置** | 不依賴模型自身記憶，所有跨階段狀態存於 Obsidian |
| **單向流動 + 回寫** | 資料依 I→P→P→O 方向處理，每階段完成後回寫 Obsidian |
| **邏輯與視覺分離** | Process 階段鎖定「對不對」，Output 階段只負責「好不好看」，兩者不可混淆職責 |
| **可追溯性** | 每份交接文件均保留原始來源與時間戳，任何結論都可回查依據 |

## 3. 各階段技術規格

### 3.1 Input 層

| 項目 | 規格 |
|---|---|
| 使用工具 | Gemini 付費版、NotebookLM |
| 輸入來源 | YouTube／TikTok／Facebook 等教學影片或圖文 |
| 處理動作 | 逐字稿擷取、章節分類、重點條列、時間戳標記 |
| 輸出格式 | 結構化文字（章節 → 條列重點 → 時間戳） |
| 品質檢核 | 來源分級：一手實戰 / 二手教學 / 純流量內容 |

### 3.2 Process 層

| 項目 | 規格 |
|---|---|
| 使用工具 | Claude |
| 核心動作 | 固定三問驗證機制（見下方） |
| 驗證機制 | 1. 費曼複述測試　2. 第一性原理拆解　3. 反例／風險點搜尋 |
| 輸出格式 | 定稿邏輯（一句話收斂結論） |
| 職責邊界 | **只處理內容正確性與邏輯嚴謹度，不涉及視覺呈現** |

### 3.3 Pipeline 層（選用）

| 項目 | 規格 |
|---|---|
| 使用工具 | Claude Code |
| 觸發條件 | 僅當需要重複性、自動化產出時啟用（單次教材整理可跳過） |
| 典型應用 | 短影音自動化生產線（Whisper→ffmpeg→Auto-Editor→ImageMagick→ElevenLabs→Pexels API） |
| 輸出格式 | 依 pipeline 需求而定（.md／.mp4／.pptx 等） |

### 3.4 Output 層

| 項目 | 規格 |
|---|---|
| 使用工具 | GPT-5.6（或其他視覺化模型） |
| 核心動作 | 依「視覺化就緒稿」產出繁中簡報／圖像 |
| 職責邊界 | **只負責視覺呈現，不可更動 Process 階段鎖定的核心論證** |
| 品質檢核 | 版面風格、字型排版是否符合既定視覺規範（見第 6 節） |

### 3.5 Obsidian 記憶層

| 項目 | 規格 |
|---|---|
| 角色 | 四個模型共用的外部記憶體，唯一事實來源 |
| 檔案格式 | Markdown (.md)，本機儲存 |
| 核心機制 | 雙向連結 (`[[筆記名稱]]`) + 關聯圖 (Graph View) |
| 目錄建議 | `00_Inbox` / `01_Process定稿` / `02_圖片上傳` / `03_GPT交接稿` |
| 附件設定 | 需優化「新附件的預設位置」，避免圖片散落根目錄 |

## 4. 標準作業程序（SOP）

### 4.1 前置條件檢查

執行任何一份素材前，先確認以下三項：

- [ ] 已取得影片逐字稿／字幕，或自行整理的重點摘要
- [ ] 已判斷主題定位：`工具教學型` / `商業策略型` / `趨勢觀點型`
- [ ] 已確認應用場景：`資策會/工研院課綱` / `CGM Coach短影音` / `EMBA教材` / `內部SOP` / `其他`

### 4.2 Step 1：Input 階段

**動作步驟：**
1. 取得影片逐字稿／字幕
2. 用 Gemini／NotebookLM 依「大主題 → 子章節 → 重點條列」分類
3. 標記時間戳，供後續回查

**輸出格式：**
```markdown
一、[章節名稱]
   - 重點1 [時間戳]
   - 重點2 [時間戳]
二、[章節名稱]
   ...
```

### 4.3 Step 2：Process 階段（固定三問，不可省略）

| 步驟 | 問題 | 目的 |
|---|---|---|
| 1 | 費曼複述：能否用自己的話講給外行人聽？ | 檢驗是否真正理解，而非複製貼上 |
| 2 | 第一性原理拆解：結論成立的前提假設是什麼？是否適用於跨界情境（生技/寵物/教學/體育）？ | 避免盲目套用他人框架 |
| 3 | 反例／風險點搜尋：是否有被忽略的限制或反對案例？ | 提升結論的穩健性 |

**必要輸出：**
- 費曼複述版本
- 第一性原理拆解
- 反例／風險點
- 最終定稿邏輯

### 4.4 Step 3：Pipeline 階段（條件觸發）

```
IF 素材需重複產出（例如每週固定教學影片轉短影音）
    → 設計 Claude Code pipeline
ELSE
    → 跳過，直接進入 Step 4
```

### 4.5 Step 4：Output 階段（視覺化交付前必答三問）

1. 最終用於什麼場合？（決定版面風格）
2. 是否有既定視覺規範需套用？（見第 6.2 節設計系統）
3. 哪些論證邏輯**絕對不可被視覺模型更動**？

## 5. Obsidian 交接文件模板

每次處理新素材時，複製以下模板並命名為 `[主題名稱]_v[版本號].md`：

```markdown
# [主題名稱]_v[版本號]

## 一、原始素材來源（Input階段）
- 來源連結/平台：
- Gemini/NotebookLM 初步重點：

## 二、驗證後結論（Process階段 - Claude）
- 費曼複述版本：
- 第一性原理拆解：
- 反例/風險點：
- 最終定稿邏輯：

## 三、待自動化事項（Claude Code階段，若需要）
- pipeline 需求：
- 輸出格式：

## 四、視覺化交付指令（Output階段 - GPT-5.6）
- 版面風格要求：
- 繁中字型/排版規範：
- 禁止事項（不可更動的核心論證）：
```

## 6. 工具分工矩陣

### 6.1 模型職責分工

| 模型 | 對應 IPPOO 層級 | 核心職責 | 不應執行的工作 |
|---|---|---|---|
| Gemini / NotebookLM | Input | 資料粗整理、多來源交叉比對 | 深度邏輯驗證 |
| Claude | Process | 邏輯推理、費曼複述、第一性原理驗證 | 最終視覺排版 |
| Claude Code | Pipeline | 自動化腳本、系統整合、重複性產出 | 內容邏輯判斷 |
| GPT-5.6 | Output | 繁中圖像／簡報最終視覺化 | 更動已驗證的論證邏輯 |

### 6.2 視覺化設計系統規範（Output 階段套用）

適用於 EMBA／MedTech 會議簡報風格：

```yaml
畫布比例: 16:9 橫式
背景: 淺藍-青綠漸層 + 玻璃擬態卡片
卡片: 24px 圓角、半透明白色面板、柔和藍色光暈邊框
色彩: 淺藍/淺綠/白漸層背景，科技藍標題，亮藍/青/金色點綴
字型:
  繁中: 思源黑體 / Noto Sans TC
  英文: Montserrat / Inter
版面原則: 一張卡片一個重點，留白充足，不可過度飽和
禁止: 深色背景、過度裝飾、資訊過載
```

## 7. 自動化擴充建議

### 7.1 Claude Code Pipeline 待辦事項

- [ ] 建立標準化 Vault 目錄結構初始化腳本
- [ ] 撰寫「Process 定稿自動回寫 Obsidian」腳本，檔名格式 `[主題]_v[版本]_[日期].md`
- [ ] 圖片附件路徑自動指向 `02_圖片上傳` 資料夾
- [ ] 撰寫「孤島筆記」掃描腳本，自動列出未建立雙向連結的關鍵詞

### 7.2 七工具 MCP 短影音產線整合點

現有工具鏈：`Whisper → ffmpeg → Auto-Editor → ImageMagick → ElevenLabs → Pexels API`

建議整合方式：Process 階段定稿的逐字稿與腳本，作為 Whisper 語音轉文字校對基準，確保短影音字幕與原始驗證邏輯一致，不因自動化流程而失真。
